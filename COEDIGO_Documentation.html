1. Project Summary

COEDIGO is a role-based academic records system for the College of Engineering. It provides login, user management, class records, subject management, grade encoding, attendance tracking, student grade viewing, audit logging, notifications, and email-based account onboarding.

The application is split into a React frontend and a PHP backend API with a MySQL database. The frontend talks to the backend through the /api path.

2. Technology Stack

    • attendance_records

9. Authentication and User Behavior

  • Login is handled by POST /api/auth/login.
  • The backend checks email, password hash, and account active status.
  • Inactive accounts are not permanently removed by default. They are soft-deactivated through is_active = 0.
  • Disabled users receive this login message:
    Account disabled. Please contact your program chair or the College of Engineering faculty office.
  • Successful login returns a token and a role-specific user profile.

10. Email Delivery After Account Creation

When an admin creates a user, the system can send a welcome email using Gmail SMTP or another SMTP server. If the admin leaves the password blank during account creation, the system generates a random temporary password.

10.1 Required Settings

Setting Description
smtp_host SMTP host, typically smtp.gmail.com
smtp_port SMTP port, usually 465 for SSL or 587 for TLS
smtp_username Full Gmail address or SMTP username
smtp_password Gmail app password or SMTP password
smtp_encryption ssl, tls, or none
mail_from_address Sender email address, usually the same as the Gmail address
mail_from_name Display name for outgoing emails
mail_reply_to Optional reply-to address

10.2 Where to Configure Mail

  • Admin panel: System Settings
  • Environment fallback values:
    • SMTP_HOST
    • SMTP_PORT
    • SMTP_USERNAME
    • SMTP_PASSWORD
    • SMTP_ENCRYPTION
    • MAIL_FROM_ADDRESS
    • MAIL_FROM_NAME
    • MAIL_REPLY_TO

10.3 Mail Delivery Notes

  • The backend uses cURL for SMTP if available.
  • If cURL is missing, it falls back to raw SMTP sockets.
  • The server must be able to reach your SMTP host, such as smtp.gmail.com.

12. Common Troubleshooting

12.1 Login Fails Even When the User Exists

  1. Confirm that the frontend can reach the backend.
  2. Check that the backend is running on 127.0.0.1:8000.
  3. Check that the frontend Vite server is running on 127.0.0.1:5173.
  4. Verify the database credentials in the backend environment.
  5. Verify that the account is active by checking users.is_active = 1.
  6. Verify that the password in the database is a valid PHP password hash.

12.2 Account Creation Succeeds in UI But Row Is Missing

  1. Check whether the frontend is talking to the same backend instance you are inspecting.
  2. Confirm that the backend is connected to the same database you are viewing in phpMyAdmin.
  3. Review backend/logs/app-YYYY-MM-DD.log for user creation errors.

12.3 Email Not Sent

  1. Verify SMTP settings in Admin > System Settings.
  2. Verify that the Gmail account has 2-Step Verification and a valid app password.
  3. Check backend logs for mail.welcome.failed or mail.smtp.response_error.
  4. Verify outbound SMTP access to smtp.gmail.com on port 465 or 587.

13. Production Deployment Guide

The cleanest production deployment is to host the frontend and backend under the same domain, with the backend exposed under the /api path.

13.1 Recommended Production Architecture

  • Frontend: static files from frontend/dist/
  • Backend: PHP application in backend/
  • Database: MySQL or MariaDB on the same server or a managed DB server
  • HTTPS: required

13.2 Recommended Production Environment Variables

Variable Purpose
DB_HOST Database host
DB_NAME Database name
DB_USER Database user
DB_PASS Database password
SMTP_HOST SMTP host
SMTP_PORT SMTP port
SMTP_USERNAME SMTP username or Gmail address
SMTP_PASSWORD SMTP password or Gmail app password
SMTP_ENCRYPTION SMTP encryption mode
MAIL_FROM_ADDRESS Sender email
MAIL_FROM_NAME Sender display name
MAIL_REPLY_TO Optional reply-to address

13.3 Build the Frontend for Production

cd C:\Users\DjMhel\Documents\COEDIGO\frontend
npm install
npm run build

Deploy the contents of frontend/dist/ to your web server.

13.4 Frontend Production API URL

If frontend and backend are served from different origins, build the frontend with VITE_API_BASE_URL set to the real backend endpoint, for example: 

VITE_API_BASE_URL=https://coedigo.example.edu/api

13.5 Apache Deployment Notes

  • Point the frontend site root to the built frontend files.
  • Expose the PHP backend under /api.
  • Keep backend/.htaccess active so non-file API routes are rewritten to backend/index.php.
  • Enable Apache rewrite support.

13.6 Example Apache Layout

DocumentRoot /var/www/coedigo/frontend/dist

/var/www/coedigo/frontend/dist         -> static frontend build
/var/www/coedigo/backend               -> PHP API application
https://your-domain/                  -> frontend
https://your-domain/api/...           -> backend/index.php router

13.7 Nginx Deployment Notes

Nginx is also valid, but you must replicate the rewrite behavior currently provided by backend/.htaccess. All API requests that are not actual files should be passed to backend/index.php.

13.8 File and Folder Permissions

  • The web server must be able to read all backend and frontend files.
  • The backend process must be able to write to backend/logs/.
  • Do not expose internal backend logs publicly over HTTP.

13.9 SSL and Security Baseline

  • Use HTTPS only in production.
  • Replace the hard-coded JWT secret in backend/middleware/AuthMiddleware.php with a production-managed secret.
  • Change the default admin password.
  • Use a database user with only the required permissions.
  • Restrict public access to phpMyAdmin or remove it from production.

13.10 Backup Strategy

  • Back up the MySQL database daily.
  • Back up the backend/logs/ directory if audit trace retention is required.
  • Back up any uploaded assets if file uploads are added later.

14. Release Procedure for Future Updates

  1. Pull the latest repository changes into a staging environment.
  2. Run database updates if the schema changed.
  3. Install frontend dependencies and rebuild the frontend.
  4. Deploy frontend/dist/ and backend PHP files.
  5. Verify login, user creation, grade views, and email delivery in staging.
  6. Deploy to production during a controlled maintenance window if needed.
  7. Monitor error logs immediately after release.
